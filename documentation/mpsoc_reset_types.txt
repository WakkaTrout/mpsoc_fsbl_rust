The following information has been gathered from https://docs.amd.com/v/u/en-US/ug1085-zynq-ultrascale-trm

This is written from the perspective from the FSBL and what it experiences due to the following resets.

1. Software Debugger Reset
    Description: Debugger wrote to BLOCKONLY_RST
    Components Reset:
    Components Preserved (including register subsets):

2. Software System Reset (a.k.a soft reset)
    Description: RESET_CTRL written to
    Components Reset:
    Components Preserved (including register subsets):

3. External System Reset
    Description: PS_SRST_B pin assert
    Components Reset:
    Components Preserved (including register subsets):

4. PS Only Reset
    Description: PMU_GLOBAL.GLOBAL_RESET written to. FSBL clears after PS-only reboot sequence
    Components Reset:
    Components Preserved (including register subsets):

5. Internal System Reset
    Description: Caused by a system error that can corrected by a reset
    Components Reset:
    Components Preserved (including register subsets):

6. Internal POR
    Description: Caused by a system error that can only be corrected by a POR reset
    Components Reset:
    Components Preserved (including register subsets):

7. External POR
    Description: PS_POR_B signal asserted
    Components Reset: PMU Processing State, CSU Processing State, PMU RAM, CSU RAM, OCM?, A53 Processing State (if running?), caches?, R5 Processing State (if running?), Mali GPU (if running?), PL (if running?), internal transcievers?, peripheral controllers?, any actual peripherals?, clocks?
    Components Preserved (including register subsets): What peripherals are preserved is system dependent and what the External POR signal resets other than the MPSoC. The FSBL may or may not assume attached peripherals are reset as well.
